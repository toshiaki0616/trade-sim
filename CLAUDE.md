# CLAUDE.md

# 基本方針
- 必ず日本語で応対してください
- 調査やデバッグにはサブエージェントを活用してコンテキストを節約してください
- 重要な決定事項は定期的にマークダウンファイルに記録してください

# コード規約
- TypeScriptを使用
- テストはVitestで書く
- コミットメッセージは日本語で簡潔に

# Core Rules (絶対遵守)
- **Language**: 
  - Thinking Process: English (for better logic)
  - Final Response: Japanese (丁寧語で回答すること)
- **Security**: 
  - `.env` ファイルやAPIキーの中身は絶対に出力しないこと。
  - 認証情報は環境変数から読み込むコードのみを提案すること。

# コーディング指針

## 命名規則
- 変数名・関数名は英語で、具体的かつ意味が伝わる名前にする
- 関数名は動詞で始める（compute_, fetch_, validate_, format_ 等）
- ブーリアン型は is_, has_, can_, should_ を接頭辞にする（例：is_valid, has_permission）
- 定数は UPPER_SNAKE_CASE（例：MAX_RETRY_COUNT）
- tmp, retval, data, foo 等の汎用名は使わない
- 単位がある値は名前に含める（例：delay_secs, size_mb）
- スコープが大きい変数ほど長く明確な名前にする

## 関数設計
- 一つの関数は一つの責務だけを持つ（Single Responsibility）
- 無関係な下位問題は別関数に切り出す
- 標準ライブラリで実現できる処理は自前実装しない
- 不要なコード・未使用コードは書かない

## 制御フロー
- ガード節（早期リターン）でネストを浅く保つ
- 条件式は肯定形を優先（disable_ssl より use_ssl）
- 複雑な条件は説明変数に代入して意味を持たせる
- 三項演算子は単純な二者択一のみに使用

## 変数管理
- const（定数）を最優先で使い、再代入を最小限にする
- 変数は使う直前に定義する（関数先頭にまとめない）
- 一度しか使わず説明にもならない中間変数は削除

## コメント
- 「なぜそうしたか（Why）」を書く。「何をしているか（What）」は書かない
- 定数の背景、判断理由、非自明な挙動への警告を書く
- ひどいコードをコメントで補わない。コード自体を改善する
- TODO/FIXME は改善箇所のマーキングに使う

## 品質チェック
- 実装後は必ずテストコードを書き、実行して動作確認する
- エッジケース（空値、null、境界値）を考慮する
- 同じコードの繰り返しがあればDRY原則で共通化する
